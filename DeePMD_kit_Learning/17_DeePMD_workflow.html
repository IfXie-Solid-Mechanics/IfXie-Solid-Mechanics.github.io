        # 曲折的势函数训练过程

        # 1. 势函数训练

        服务器的机器的显存不能支持deepmd-kit的训练，在scnet上进行了测试，发现无法显存接近70G，因此也训练失败。

        我尝试过用两个节点开更多dcu来支持gpu版本的运行，但是两个节点的通讯是一个更大的技术难题，因此在scnet上的gpu版本训练计划作废。

        但是，我们后来在scent上使用cpu版本成功进行了训练，训练时间为1天半。

        # 2. 势函数的后处理

        # 2.1 势函数的freeze

        这一步无论是在超算上用cpu版本来操作，还是用本地服务器的gpu版本来操作都是没问题的。

        并且因为后续的compress过程过于曲折，因此我们在这一阶段也直接尝试使用“完成freeze单没有被compress的模型”进行了lammps模拟，发现对于32000原子的体系需要大约4小时才能跑1000个steps，效率极低。

        # 2.2 势函数的compress陷入困境

        在最初尝试了在scnet上的cpu版本进行compress，但是发现compress失败，原因是递归过深：

        RecursionError: maximum recursion depth exceeded in comparison
        1

        我顺手在GitHub上的issue上查找了一下，找到了一个2024年7月的issue，但是是gpu版本的问题，并且是版本兼容的相关问题（第一个是新版本的小bug，第二个是dp版本和tf版本不兼容，或者说他换了一个tf版本就跑通了）

        后来尝试在本地进行了compress操作，但是发现本地服务器和scnet上的不兼容问题暴露了：本地服务器是v2.2.10，而scnet上的版本是v3.0.0，v3.0.0在model部分存在更多的描述性信息，这些信息无法被本地v2.2.10读取，直接报错。

        而scent的gpu版本几乎不支持compress这个操作，至少我直接sbatch上去之后没有正常运行，报错信息是：

        File "/opt/deepmd-kit_v3.0.0b4_pytorch/lib/python3.8/site-packages/deepmd/tf/env.py", line 84, in <module>
    import tensorflow.compat.v1 as tf
ModuleNotFoundError: No module named 'tensorflow'
        1
        2
        3

        # 2.3 势函数的compress尝试突困

        根据上面的信息，我们可以尝试将本地的deepmd-kit升级成为v3.0.0版本（其实，我是直接安装了一个新的v3.0.0版本，老版本也暂时留着）

        首先，去南京大学的镜像网站扒deepmd-kit的两个release文件，然后把它们合并一下再执行，建议在home目录执行：

        cat deepmd-kit-3.0.0-cuda126-Linux-x86_64.sh.0 deepmd-kit-3.0.0-cuda126-Linux-x86_64.sh.1 > deepmd-kit-3.0.0-cuda126-Linux-x86_64.sh
sh deepmd-kit-3.0.0-cuda126-Linux-x86_64.sh -b
        1
        2

        然后home目录中应该会出现一个deepmd-kit目录，之后需要激活一下刚刚装好的环境：

        source ~/deepmd-kit/bin/activate
        1

        此时即可查看dp的版本：

        dp --version
#DeePMD-kit v3.0.0
        1
        2

        然后就可以正常地使用dp去compress了，而这一次直接一次成功。（感觉新版本的显存分配也智能了很多，后续可以尝试一下）

        # 3. 势函数用于合金的relax过程

        这里回到scnet上的gpu版本（主要是优化做得太好了），已经能够正常使用CrFeCoNiPd-compress.pb文件了，3分钟运行了2000steps，至少技术性上这个问题被完全解决。

        即使用scnet的cpu版本进行训练，在本地进行训练结果的后处理，将势函数模型放在scnet上运行lammps，这是目前实践下来能够走通的一套完整工作流。
