    # 1.4 第五行之后是Nelements sections

    LAMMPS官方文档的写法是

    line 1 = atomic number, mass, lattice constant, lattice type (e.g. FCC)
embedding function F(rho) (Nrho values)
density function rho(r) (Nr values)
    1
    2
    3

    也就是对于元素A的F(ρ),ρ(r)打表。

    理论上一列就可以了，但是我也看到了很多五列的，LAMMPS官方文档对此也没有说明，我猜测或许是取平均值？

  -1.18796003752840065e-01   -1.32884412197021823e-01   -1.46331083774124904e-01   -1.58931976672798347e-01   -1.70783852107104811e-01
    1

    # 1.5 最后是对势部分

    LAMMPS的官方文档写法是

    Nr values for each pair potential phi(r) array are listed for all i,j element pairs in the same format as other arrays.
    1

    并且由于ϕij=ϕji打表只打i大于或等于j的部分，具体可以看LAMMPS官方文档。

    # 2. 开发思路

    首先，对于ϕ(r)的拟合，可以通过VASP第一性原理计算得到一个比较像的方程，用于初步拟合，然后再不断微调；

    其次，对于ρ(r)的拟合和F(ρ)的拟合是我们开发利用DeePMD-kit的关键：

    对于ρ(r)，张老师认为可以用神经网络中的embedding模型来进行训练；

    而后的F(ρ)则也可以通过深度网络模型来训练。

    # 2.1 对于ρ(r)的开发思路

    这里我认为直接使用embedding模型是不妥的，因为在DeePMD-kit中的embedding模型是将截断半径内的所有rij映射为一个Di向量。

    而在LAMMPS的需求中，输入信息只有一个距离信息rij，输出信息只有一个电荷密度信息ρj。

    因此，我认为使用DNN网络来进行拟合也许效果会更好。但是如何获得监督学习的数据还需要进一步地分析。

    # 2.2 对于F(ρ)的开发思路

    这里我同样认为可以使用DNN模型来进行开发，并且对于这一部分进行监督学习是有可行性的。

    utils目录通常是用来存放各种辅助工具和实用函数的地方，目的是提高代码复用性、组织性和可维护性。

    tf目录有两种可能：可能性比较大的是这里与Tensorflow有极大关联，还有一种可能是临时文件夹（在DeePMD-kit中是前者）

    backend目录一般是偏向于服务器端的操作，比如提供API接口之类的。
